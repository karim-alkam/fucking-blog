      <p>the <strong>Modbus</strong> protocol is the standard of networking in industrial automation it was developed in 1979 by <strong>Modicon</strong> for there PLCs but it became an open <strong>standard</strong> any manufacturer can use in there devices    </p>
<h2 id="why-modbus">Why Modbus</h2>
<p>Modbus is on the application layer of the network, the meaning is it's job is to organize and structure data exchange<br>it's continued success stems from these three main factors:</p>
<ul>
<li>Open and free: It is <strong>openly published</strong> and <strong>royalty-free</strong>, making it easy for any manufacturer to adopt.</li>
<li>It's relatively easy to deploy and maintain compared to newer, more complex industrial protocols.</li>
<li><strong>Versatility:</strong> It places few restrictions on data format, making it flexible for everything from reading a single relay status to logging complex measurements.</li>
</ul>
<p>It fundamentally operates on a <strong>Client/Server</strong> principle (formerly Master/Slave), where a central <strong>Client/Master</strong> (e.g., a supervisory computer, data logger) initiates a transaction to request data or command an action from a <strong>Server/Slave</strong> (e.g., a sensor or PLC).</p>
<h2 id="the-two-dominant-flavors-rtu-vs-tcp">The Two Dominant Flavors: RTU vs. TCP</h2>

<h3 id="the-tcpip-differentiator-transaction-id">The TCP/IP Differentiator: Transaction ID</h3>
<p>Modbus TCP adds an <strong>MBAP Header</strong> (Modbus Application Protocol Header) before the core message. The most important field in this header is the <strong>Transaction ID</strong> (2 bytes).</p>
<p>This ID is used to <strong>match asynchronous requests with their responses</strong>. If a client sends three requests quickly, the server's responses might arrive out of order, but the client uses the Transaction ID to correctly pair the incoming data to the original query.</p>

<p>Every Modbus communication relies on two core data structures: the PDU and the ADU.</p>
<h3 id="pdu-protocol-data-unit">PDU (Protocol Data Unit)</h3>
<p>The PDU is the <strong>core message content</strong>, independent of the physical communication method.<br>$$PDU = Function Code + Data$$</p>
<ul>
<li><strong>Function Code (1 Byte):</strong> This is the command. It tells the server <em>what</em> to do. Examples include 0x01 (Read Coils) or 0x03 (Read Holding Registers). If the server encounters an error, it responds with an <strong>Exception Function Code</strong> (Original Code + 0x80) plus an Exception Code explaining the error.</li>
<li><strong>Data:</strong> Includes the <strong>Starting Address</strong> (where to start the operation, 2 bytes) and the <strong>Quantity</strong> (how many items to read/write).</li>
</ul>
<h3 id="adu-application-data-unit">ADU (Application Data Unit)</h3>
<p>The ADU is the <strong>complete, transmitted frame</strong> that wraps the PDU with transport-specific fields.</p>
<ul>
<li><p><strong>Modbus RTU ADU</strong>: $Slave Address + PDU + CRC$</p>
<ul>
<li><strong>Slave Address (1 byte)</strong>: Identifies the specific server (slave) device on the serial line that the client (master) wants to communicate with. This is the device's unique physical address on the bus. </li>
<li><strong>PDU (Up to 253 bytes)</strong>: <strong>Protocol Data Unit</strong> (Function Code + Data). This is the core instruction or response, which is identical in both RTU and TCP versions.</li>
<li><strong>CRC (2 bytes)</strong>: <strong>Cyclic Redundancy Check</strong>. This is a dedicated checksum used for robust <strong>error checking</strong>. The receiving device calculates the CRC for the incoming data and compares it to the received CRC value to ensure data integrity.</li>
</ul>
</li>
<li><p><strong>Modbus TPC ADU</strong>: $MBAP Header + Unit ID + PDU$</p>
<ul>
<li><strong>MBAP Header (6 bytes)</strong>: <strong>Modbus Application Protocol Header</strong>. This header is essential for network communication and is broken down into three fields:<ul>
<li><strong>Transaction ID (2 bytes)</strong>: A sequential tag used by the client to <strong>match requests to their corresponding responses</strong> when they arrive.</li>
<li><strong>Protocol ID (2 bytes)</strong>: Always 0x0000, identifying the Modbus protocol.</li>
<li><strong>Length (2 bytes)</strong>: Specifies the number of bytes that follow in the rest of the frame (Unit ID + PDU).</li>
</ul>
</li>
<li><strong>Unit ID (1 byte)</strong>: Used to <strong>identify the specific device</strong> when Modbus TCP is used to talk to an RTU network through a gateway. It acts as the slave address on the internal serial segment.</li>
<li><strong>PDU (Up to 253 bytes)</strong>: <strong>Protocol Data Unit</strong> (Function Code + Data). This is the core instruction or response, identical to the RTU version.</li>
</ul>
</li>
</ul>
<p>The <strong>ADU</strong> is what the client actually constructs and sends, and it carries the PDU across the wire to complete the transaction.</p>
